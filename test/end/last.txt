  🔍 Project Understanding
  
  F1 is a VS Code extension that embeds a multi-session CLI launcher
  panel. The entry point (src/extension.ts) registers a single
  WebviewViewProvider (CliHubViewProvider). Two HTML phases live inside
  that single panel view: a fuzzy-search launcher (index.ts + index.html)
  that lets users pick a CLI, and a full terminal UI (terminal.ts +
  xterm.js) once an agent is chosen. PTY processes are isolated in a
  subprocess (pty-host.ts) spawned per session. All communication crosses
  two message buses: webview ↔ host (postMessage) and host ↔ pty-host
  (Node IPC). Host-side tools (translation, TTS, spell-check, image prep)
  live in tools-cli-core/. The project compiles four esbuild targets and
  has no test suite.

🟡 Debt/Risks

1. Extension mode detection contains redundant logic and an unreachable fallback path, making future VS Code changes harder to detect.

2. Workspace paths are shortened to project names, which can display incorrect locations when multiple workspaces share the same folder name.

3. File listing generates synthetic directory entries that may create duplicates or confusing results in the @-mention picker.

4. CliHubViewProvider concentrates too many responsibilities in a single class, increasing coupling and long-term maintenance costs.

5. HTML template replacement relies on manual `replace()` calls, which may miss repeated placeholders and cause subtle rendering issues.

🟢 Suggestions (optional)

1. Make `_extensionContext` required instead of optional to simplify the codebase and remove defensive fallbacks.

2. Move `serializeJsonForHtmlScript()` into a shared utility module to reduce the size and responsibility of `main.ts`.

3. Keep `extension.ts` unchanged; its implementation is already clean and follows VS Code best practices.
