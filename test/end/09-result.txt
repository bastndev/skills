  🟡 Debt/Risks
  
  1. `getAgentSlug` is duplicated verbatim in `src/clihub/index.ts` and
     `src/clihub/webview/ui/panel-terminal/terminal.ts` — any new agent
  requires
     two edits and the functions can silently diverge (index.ts returns
  `''`
     on no-match; terminal.ts returns `'default'`).
  
  2. `main.ts` has grown to ~500 lines, mixing HTML generation, message
  dispatch,
     voice, translation, spellcheck, workspace listing, and clipboard
  into a
     single class — high change frequency and cognitive load for any
  future
     feature work.
  
  3. `launcherAgents` (main.ts) and `cliAgents` (agents.ts) are parallel
  arrays
     that must be kept in sync manually; adding a new agent requires
  edits in
     both plus `getAgentSlug` in two files, the icon folder, and
  cli-installers —
     four separate locations with no compile-time guard if one is
  missed.
  
  4. `_handleWorkspaceListFiles` builds the directory list by deriving
  parent
     paths from file paths in JS rather than using
  `vscode.workspace.fs.readDirectory`,
     causing unnecessary work and making the dir list unreliable for
  empty directories.
  
  5. Translation and image-save requests in `main.ts` use `any` casts
     (`ctx ... as any`) and absence of an `_extensionContext` guard on
     `preparePromptForCLI` — if the provider is ever constructed without
  a context
     the call reaches `vscode.workspace.fs` with
  `undefined.globalStorageUri`
     and throws at runtime.
  
  🟢 Suggestions (optional)
  
  1. The `CliHubMessage` type in `main.ts` uses all-optional fields,
  making every
     handler do its own field-type-narrowing; a discriminated union
  (matching the
     already-typed `ClientMessage` in terminal.ts) would remove that
  boilerplate.
  
  2. `ARCHITECTURE.md` and `README.md` are empty files — filling them in
  would
     significantly lower onboarding cost given how non-obvious the
  two-phase
     webview and four build-target setup is.
  
  3. The `tsconfig.json` has `noImplicitReturns`,
  `noFallthroughCasesInSwitch`,
     and `noUnusedParameters` commented out — enabling them would catch
  classes
     of bugs for free during type-check.

  Reason: The four-target build split, PTY subprocess isolation, and
  host/webview message boundary are sound architectural decisions.
  Issues are
  internal to existing modules — duplicated logic, an oversized class,
  and
  parallel data tables — not structural problems.
  
  Adjustments:
  - Extract a shared `getAgentSlug` utility into a file importable by
  both
    launcher and terminal bundles, or consolidate the slug map in
  `agents.ts`.
  - Break `CliHubViewProvider` into focused handler modules (voice,
  translation,
    workspace, prompt) that the provider delegates to.
  - Merge `launcherAgents` and `cliAgents` (or derive one from the
  other) so
    a single source of truth drives both the launcher UI and the PTY
  session.
