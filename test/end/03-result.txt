  🔍 Project Understanding
  
  F1 is a VS Code extension that embeds a multi-session CLI launcher
  panel. The entry point (src/extension.ts) registers a single
  WebviewViewProvider (CliHubViewProvider). Two HTML phases live inside
  that single panel view: a fuzzy-search launcher (index.ts + index.html)
  that lets users pick a CLI, and a full terminal UI (terminal.ts +
  xterm.js) once an agent is chosen. PTY processes are isolated in a
  subprocess (pty-host.ts) spawned per session. All communication crosses
  two message buses: webview ↔ host (postMessage) and host ↔ pty-host
  (Node IPC). Host-side tools (translation, TTS, spell-check, image prep)
  live in tools-cli-core/. The project compiles four esbuild targets and
  has no test suite.

  ⚠️ Findings
  
  🔴 Confirmed Bugs
  
  - `src/clihub/main.ts:_handleWorkspaceListFiles`
    Problem: `isDirectory` is hardcoded to `false` for every file entry,
  then directories are
    inferred and appended separately — but the inferred directories may
  produce duplicates if
    any deeply nested file shares an ancestor already listed.
    Impact: @-file mention picker can surface duplicate directory
  entries, breaking UX.
    Recommendation: Remove the redundant directory-inference loop; use
  only the `vscode.workspace.findFiles`
    results (all files, no directories needed) or separate the two
  queries cleanly. — [non-critical → phase 3]
  
  🟠 Risks
  
  - `src/clihub/main.ts:_getHtmlForWebview`
    Problem: HTML template is populated with `String.replace()` using
  literal `${key}` tokens.
    A `selectedAgent` or workspace path value containing `${}` syntax
  could silently corrupt HTML.
    Impact: Maliciously crafted workspace path or agent label could
  inject into the webview HTML.
    Recommendation: Switch to a small explicit interpolation function
  that escapes every substituted value
    (the `escapeHtml` helper already exists in `webview.ts`; centralize
  and reuse it).
  
  -
  `src/clihub/webview/core/terminal-cli/session-manager.ts:startPtyHost`
    Problem: `CLIHUB_NODE_PATH` is consumed from `process.env` without
  validation and passed
    directly to `childProcess.spawn` as the executable.
    Impact: If the environment variable is poisoned (e.g. via a malicious
  `.env` in the workspace),
    an arbitrary binary is executed in the extension host context.
    Recommendation: Validate that the path resolves to an executable file
  before trusting it,
    or document that only the extension itself should set this variable.
  
  - `src/clihub/webview/core/terminal-cli/installation.ts:commandExists`
    Problem: The Unix branch builds a shell string with single-quote
  escaping for user-supplied
    `command` values (`command.replace(/'/g, "'\\''")`). This is a
  partial mitigation — while
    correct for simple cases, it passes the string to `sh -lc`, so any
  bypass in the escaping
    reaches a shell.
    Impact: A malicious agent `command` value (attacker-controlled
  `agents.ts`) could achieve
    code execution during the install-check.
    Recommendation: Use `which`/`command -v` via `spawn` with an args
  array (no shell) so the
    command name is passed as a direct argument, never interpolated.
  
  🟢 Refactoring Opportunities
  
  - `src/clihub/index.ts` + `src/clihub/main.ts:getAgentSlug`
  Problem: `getAgentSlug` is defined identically in both files (same
  if-chain, same return values).
  Impact: Any future agent addition requires changing two places; they
  have already drifted
  slightly (one returns `'default'`, the other returns `''` for the
  fallback).
  Recommendation: Extract to `src/clihub/webview/shared/agent-slug.ts`
  and import in both build targets.
  
  - `src/clihub/main.ts:launcherAgents` +
  `src/clihub/webview/core/terminal-cli/agents.ts:cliAgents`
  Problem: Agent data is split across two arrays in different files with
  no shared source of truth.
  Each agent addition requires updating both arrays plus both
  `getAgentSlug` functions (4 changes minimum).
  Impact: Easy to forget one; already evident from Cursor having `args:
  ['agent']` in `cliAgents`
    with no corresponding note in `launcherAgents`.
  Recommendation: Merge into a single `agents` definition that carries
  all fields
  (label, command, args, aliases, iconFile, darkIcon, lightIcon), then
  derive the two current shapes from it.
  
  - `src/clihub/main.ts:resolveWebviewView`
  Problem: The message handler is a large flat if-chain (~60 lines of `if
  (message.type === ...)` blocks).
  Impact: Hard to scan, test, or extend; each new message type grows this
  function further.
  Recommendation: Extract a `handleWebviewMessage` method (or a dispatch
  map) to keep `resolveWebviewView`
    focused on setup.
  
  - `src/clihub/webview/ui/panel-terminal/terminal.ts`
  Problem: Module-level mutable state (`sessions`, `terminals`,
  `bootSkeletons`,
  `pendingPromptTranslations`, etc.) is spread across 14+ `let`/`const`
  declarations at the top.
  All async message flows are handled by a single
  `window.addEventListener('message', ...)`.
  Impact: Makes the module hard to reason about and would require
  significant rework to unit-test.
  Recommendation: Group related state + handlers into small objects or
  classes
    (e.g. `SessionRegistry`, `PromptBridge`). No behavior change needed.
  
  🟡 Technical Debt
  
  - `src/clihub/main.ts:_handleWorkspaceListFiles`
    Problem: Has a TODO comment referencing future NLS support ("TODO:
  future multi language nls.json")
    in installation.ts; `_handleWorkspaceListFiles` also has a redundant
  `isDirectory = false` comment
    that marks known incomplete design.
    Impact: Raises future development cost when i18n or proper file-type
  handling is needed.
    Recommendation: Track as a follow-up issue; no code change needed
  now.
  
  - `src/clihub/webview/core/terminal-cli/session-manager.ts`
    Problem: `maxBufferLength` (240,000) is a magic number duplicated in
  both `session-manager.ts`
    and `terminal.ts` (`maxWebviewBufferLength`).
    Impact: Future changes to buffer size require coordinating two files;
  mismatches cause subtle
    truncation differences between host and webview state.
    Recommendation: Export a shared constant from a shared types file.
  
  - No test suite
    Problem: The project has zero tests (no .test.ts, no spec files, no
  test runner configured).
    Impact: Any refactor or bug fix is validated manually only;
  regressions are invisible until
    a user reports them.
    Recommendation: Document as known risk; do not create test
  infrastructure as part of this refactor.
  
  - `package.json` — `bun run` in AGENTS.md vs `npm run` in package.json
  scripts
    Problem: AGENTS.md documents `bun run compile` but `package.json`
  defines npm scripts
    (no bun-specific config). Both work since bun can run npm scripts,
  but the discrepancy
    is confusing for new contributors.
    Impact: Minor contributor confusion; no runtime effect.
    Recommendation: Standardize the documented command to `npm run` or
  add a `bunfig.toml`.

  🏗️ Architecture Decision
  
  📐 Improvement Refactor Only
  
  Reason: The two-phase webview, PTY subprocess isolation, and
  message-bus boundaries
  are intentional and well-suited to VS Code's extension model. The
  structure is sound;
  what needs attention is data-duplication between agent definitions, the
  duplicated
  getAgentSlug, the flat message handler, and two security-adjacent
  string-handling
  issues — all internal cleanup with no architectural boundary changes
  needed.
  
  Adjustments:
  - Merge agent data into a single source of truth (eliminates 4-touch
  agent additions).
  - Extract shared getAgentSlug to a shared module (eliminates silent
  drift).
  - Extract webview message dispatch from resolveWebviewView (reduces
  cognitive load).
  - Fix the shell-injection risk in commandExists (replace sh -lc with
  spawn + args array).
