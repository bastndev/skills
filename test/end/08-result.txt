  🟡 Debt/Risks
  
  - src/clihub/main.ts:CliHubViewProvider._getAgentLaunchExtensionMode
  
    Problem: Three if branches return identical extensionMode values in
  the same order VS Code evaluates them internally; the final return
  'unknown' is unreachable in practice but adds noise and could silently
  hide a future new mode.
    Impact: Dead code path; any new ExtensionMode value added by VS Code
  will silently fall to 'unknown' without a type error.
    Recommendation: Replace the if-chain with a const lookup map
  (Map<vscode.ExtensionMode, AgentLaunchExtensionMode>) so adding a new
  mode produces a TS error immediately.
  
  - src/clihub/main.ts:CliHubViewProvider._getWorkspacePath
  
    Problem: Always returns ~/${projectName}, stripping the real
  absolute path. If two projects share the same basename (e.g. both
  named src), the terminal CWD display will silently show the wrong
  path.
    Impact: Misleading CWD displayed in terminal status bar; harder to
  debug multi-root workspace setups.
    Recommendation: Return the real workspace path, or at minimum
  document the intentional shortening.
  
  - src/clihub/main.ts:CliHubViewProvider._handleWorkspaceListFiles
  
    Problem: The isDirectory = false comment says "findFiles only
  returns files", yet the function then constructs dirs from those file
  paths and re-injects them as entries with isDirectory: true. This
  creates duplicate path entries (file + synthetic dir) and skips any
  empty directories.
    Impact: The @-mention file picker may show duplicate or phantom
  entries, confusing the user.
    Recommendation: Separate file listing from directory listing, or
  document that synthetic dirs are intentional and deduplicate entries.
  
  - src/clihub/main.ts:CliHubViewProvider (class level)
  
    Problem: CliHubViewProvider handles 12+ distinct message types, HTML
  generation (launcher + agent), HTML template string replacement,
  workspace file listing, workspace skill listing, spellcheck,
  translation, voice, clipboard, CSP assembly, and session lifecycle —
  all in one 500-line class.
    Impact: High coupling makes future changes brittle; adding a new
  tool feature requires modifying the already large resolveWebviewView
  handler.
    Recommendation: Extract message handlers into a dedicated
  CliHubMessageRouter and HTML builders into a CliHubHtmlBuilder; keep
  the provider as thin coordinator.
  
  - src/clihub/main.ts:CliHubViewProvider._getHtmlForWebview
  
    Problem: Uses fs.readFileSync + manual html.replace('${key}', value)
   for HTML templating. If the template ever contains a key more than
  once (other than ${nonce} which already uses replaceAll), the second
  occurrence silently goes unreplaced.
    Impact: Subtle rendering bugs if template keys appear more than
  once; replaceAll is used inconsistently (only for nonce).
    Recommendation: Use replaceAll for all template key replacements for
  consistency and safety.
