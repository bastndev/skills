  🔍 Project Understanding
  
  F1 is a VS Code extension (TypeScript) that embeds AI CLI tools in a
  webview panel. Entry point: src/extension.ts → CliHubViewProvider in
  src/clihub/main.ts. Two webview phases: Phase 1 is a fuzzy-search
  launcher (index.ts / index.html); Phase 2 replaces it with an xterm.js
  terminal layout. A PTY host subprocess (pty-host.js) isolates node-pty
   from the renderer via Node.js IPC. Tools panel in the host handles
  translation (MyMemory/Google), TTS (piper), spellcheck, and
  image-attachment preparation. Four esbuild targets produce Node.js and
  browser IIFE bundles. No test suite.

  ⚠️ Findings
  
  🔴 Confirmed Bugs
  
  - `src/clihub/main.ts:_handleWorkspaceListFiles`
    Problem: `isDirectory` is always hardcoded `false` for all file
  entries before directories
    are re-injected from a separate `dirs` Set. Any consumer relying on
  the `isDirectory`
    flag for actual files will receive incorrect data.
    Impact: File-picker in the prompt @-mention will misclassify all
  files as non-directories,
    but the separately-built directory entries are correct — partial
  data corruption.
    Recommendation: Remove the const `isDirectory = false` assignment
  and derive it per entry.
    — non-critical → phase 1
  
  🟠 Risks
  
  - `src/clihub/main.ts:_getHtmlForWebview`
    Problem: `html.replace('${workspacePath}', ...)` uses a plain string
  replacement on raw HTML.
    If `this._getWorkspacePath()` ever returns a string containing `${`
  (unusual but possible
    on non-standard workspace names), subsequent replacements could
  silently corrupt the HTML.
    Impact: Template substitution ordering is fragile; a malformed path
  could break the launcher.
    Recommendation: Switch to a single-pass replacement map or use regex
  with the `g` flag
    consistently for all placeholders (as already done for `${nonce}`).
  
  - `src/clihub/main.ts:resolveWebviewView`
    Problem: `webviewView.webview.html = await
  this._getHtmlForWebview(...)` is called inside
    `onDidReceiveMessage` for `cli.*` messages (the
  `closed-last-session` branch), but this
    is an async operation with no error boundary. A rejection here is
  silently swallowed.
    Impact: If `_getHtmlForWebview` throws (e.g. `index.html` missing
  from dist), the webview
    goes blank with no user feedback.
    Recommendation: Wrap in a try/catch and post an error or show a VS
  Code notification.
  
  -
  `src/clihub/webview/core/terminal-cli/session-manager.ts:startPtyHost`
    Problem: `process.env.CLIHUB_NODE_PATH || 'node'` — the fallback
  `'node'` relies on
    `$PATH` resolution, which in an Extension Host may differ from what
  the user expects,
    especially on macOS where the shell PATH isn't inherited by VS Code
  processes.
    Impact: PTY host fails to start silently on some macOS/nvm setups;
  user sees a generic
    error without guidance.
    Recommendation: Use `process.execPath` as the primary fallback (the
  currently-running
    Node binary is always resolvable) before falling back to `'node'`.
  
  🟢 Refactoring Opportunities
  
  - `src/clihub/main.ts:CliHubViewProvider`
    Problem: The class is ~500 lines and handles webview lifecycle, HTML
  generation, all
    message routing (10+ message types), translation, voice, spellcheck,
  workspace queries,
    and agent launch confirmation in a single file.
    Impact: Any new feature touches this file; high cognitive load;
  unit-testing impossible.
    Recommendation: Extract message handlers into a dedicated
  `MessageRouter` or split into
    focused host modules (voice, translate, workspace, prompt), each
  receiving `webview` as
    a parameter. Keep `CliHubViewProvider` as a thin coordinator.
  
  - `src/clihub/main.ts` / `src/clihub/index.ts`
    Problem: `launcherAgents` in `main.ts` and `cliAgents` in
  `agents.ts` are two parallel
    arrays that must stay in sync manually (label, icon, installed flag,
  aliases). The
    `getAgentSlug` function is duplicated in both `index.ts` and
  `terminal.ts`.
    Impact: Adding or renaming an agent requires edits in 4+ places;
  easy to diverge.
    Recommendation: Merge icon/alias/slug metadata into a single source
  of truth in `agents.ts`;
    derive both arrays from it.
  
  - `src/clihub/main.ts:_handleWorkspaceListFiles`
    Problem: The function manually reconstructs a directory tree from
  file URIs using a `Set`
    and `path.dirname` loop, re-inserting directories as fake entries
  with `isDirectory: true`.
    This is ~30 lines of fragile logic for a common task.
    Impact: Maintenance overhead; the `isDirectory = false` bug stems
  directly from this
    complexity.
    Recommendation: Return files only (no directory entries), or use
  `vscode.workspace.fs`
    `readDirectory` recursively if directory entries are genuinely
  needed.
  
  -
  `src/clihub/webview/core/terminal-cli/session-manager.ts:handleMessage
  `
    Problem: `switch` routes message types but `cli.ready` is handled
  twice — once here
    (calls `postState`) and once in the outer `resolveWebviewView`
  handler (calls `attach`
    + `warmSpellchecker` + `createSession`). The outer handler returns
  early before
    `handleMessage` is ever called for `cli.ready`, so the switch-case
  is dead code.
    Impact: Misleading; a maintainer may trust it and remove the outer
  handler.
    Recommendation: Remove the unreachable `cli.ready` case from the
  switch statement.
  
  🟡 Technical Debt
  
  - `src/clihub/webview/core/terminal-cli/installation.ts:commandExists`
    Problem: The Unix `command -v` check uses shell string interpolation
  with a manual
    single-quote escape (`'${command.replace(/'/g, "'\\''")}'`). While
  functional, it
    relies on sh being available and the escape being correct for all
  edge cases.
    Impact: Command names with unusual characters could bypass the
  escape; spawning a shell
    per check adds latency (mitigated by the fact it's called only on
  install-check).
    Recommendation: Use `which` or `where` directly as the spawned
  binary (no shell
    indirection) or a dedicated `which` npm package if already used
  elsewhere.
  
  - `src/clihub/main.ts`
    Problem: Comment `// TODO: future multi language nls.json.` in
  `installation.ts` and
    the hardcoded ES→EN default translation direction (`from:
  message.from || 'es'`) indicate
    planned i18n work that has not been tracked or prioritized.
    Impact: Hardcoded Spanish-first assumption; if voice/translation
  features are extended,
    locale detection needs to be addressed consistently.
    Recommendation: Track as a backlog item; for now, consider reading
  VS Code's display
    language (`vscode.env.language`) to set a better default.

  🏗️ Architecture Decision
  
  📐 Improvement Refactor Only
  
  Reason: The overall architecture is sound — clean separation between
  extension host,
  webview, and PTY subprocess; typed message protocol; proper CSP;
  modular tools under
  tools-cli-core. The main pressure point is CliHubViewProvider
  accumulating too many
  responsibilities and two parallel agent-definition arrays that must be
  kept in sync.
  No structural reorganization is needed.
  
  Adjustments:
  - Consolidate agent metadata (icons, aliases, slugs) into agents.ts as
  a single source of truth.
  - Extract message handlers from CliHubViewProvider into focused host
  modules.
  - Fix the isDirectory bug and dead cli.ready switch case.
