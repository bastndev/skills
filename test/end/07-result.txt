🏗️ Architecture Decision

📐 Improvement Refactor Only

Reason: The core architecture (two-phase webview, PTY isolation in a separate Node child with IPC, clear message protocol boundaries, separation of launcher vs. terminal UI bundles, and best-effort tools on the host) is intentional, sound for the VS Code webview + native-pty constraints, and already delivers good isolation and UX. The highest-leverage problems are duplication of agent data and the getAgentSlug helper, plus concentration of concerns in one provider class — classic maintainability debt, not fundamental structural failure. No need to change build targets, message flow, PTY model, or the two-phase swap.

Adjustments:
• Consolidate the three agent lists (cliAgents, launcherAgents, cliInstallers) + slug logic into one registry in or near agents.ts; derive everything else (including the launcher models and allowed set) from it.
• Extract shared (non-DOM) pieces that must cross the launcher/terminal bundle boundary (at minimum getAgentSlug + agent metadata types) so the documented "add a CLI" process touches one place.
• Light decoupling inside CliHubViewProvider (e.g. extract message handlers or a small router) without altering the webview ownership or session lifetime model.
• Keep all files in place; improve internal boundaries and eliminate duplication. Preserve exact current runtime behavior and public contracts.
